    `Promise`s are important building blocks for asynchronous operations in JavaScript. You may think that promises are not so easy to understand, learn, and work with. And trust me, you are not alone!

    Promises are challenging for many web developers, even after spending years working with them.

    In this article, I want to try to change that perception while sharing what I've learned about JavaScript Promises over the last few years. Hope you find it useful.

    # **What is a Promise in JavaScript?**

    AÂ `Promise`Â is a special JavaScript object. It produces a value after anÂ `asynchronous`Â (aka, async) operation completes successfully, or an error if it does not complete successfully due to time out, network error, and so on.

    Successful call completions are indicated by theÂ `resolve`Â function call, and errors are indicated by theÂ `reject`Â function call.

    You can create a promise using the promise constructor like this:

    ```jsx
    let promise = new Promise(function(resolve, reject) {
        // Make an asynchronous call and either resolve or reject
    });
    ```

    In most cases, a promise may be used for an asynchronous operation. However, technically, you can resolve/reject on both synchronous and asynchronous operations.

    # **Hang on, don't we haveÂ `callback`Â functions for async operations?**

    Oh, yes! That's right. We haveÂ `callback`Â functions in JavaScript. But, a callback is not a special thing in JavaScript. It is a regular function that produces results after anÂ `asynchronous`Â call completes (with success/error).

    The word 'asynchronous' means that something happens in the future, not right now. Usually, callbacks are only used when doing things like network calls, or uploading/downloading things, talking to databases, and so on.

    WhileÂ `callbacks`Â are helpful, there is a huge downside to them as well. At times, we may have one callback inside another callback that's in yet another callback and so on. I'm serious! Let's understand this "callback hell" with an example.

    # **How to Avoid Callback Hell â€“ PizzaHub Example**

    Let's order a Veg Margherita pizza ðŸ• from the PizzaHub. When we place the order, PizzaHub automatically detects our location, finds a nearby pizza restaurant, and finds if the pizza we are asking for is available.

    If it's available, it detects what kind of beverages we get for free along with the pizza, and finally, it places the order.

    If the order is placed successfully, we get a message with a confirmation.

    So how do we code this using callback functions? I came up with something like this:

    ```jsx
    function orderPizza(type, name) {

        // Query the pizzahub for a store
        query(`/api/pizzahub/`, function(result, error){
           if (!error) {
               let shopId = result.shopId;

               // Get the store and query pizzas
               query(`/api/pizzahub/pizza/${shopid}`, function(result, error){
                   if (!error) {
                       let pizzas = result.pizzas;

                       // Find if my pizza is availavle
                       let myPizza = pizzas.find((pizza) => {
                           return (pizza.type===type && pizza.name===name);
                       });

                       // Check for the free beverages
                       query(`/api/pizzahub/beverages/${myPizza.id}`, function(result, error){
                           if (!error) {
                               let beverage = result.id;

                               // Prepare an order
                               query(`/api/order`, {'type': type, 'name': name, 'beverage': beverage}, function(result, error){
                                  if (!error) {
                                      console.log(`Your order of ${type} ${name} with ${beverage} has been placed`);
                                  } else {
                                      console.log(`Bad luck, No Pizza for you today!`);
                                  }
                               });

                           }
                       })
                   }
               });
           }
        });
    }

    // Call the orderPizza method
    orderPizza('veg', 'margherita');
    ```

    Let's have a close look at theÂ `orderPizza`Â function in the above code.

    It calls an API to get your nearby pizza shop's id. After that, it gets the list of pizzas available in that restaurant. It checks if the pizza we are asking for is found and makes another API call to find the beverages for that pizza. Finally the order API places the order.

    Here we use a callback for each of the API calls. This leads us to use another callback inside the previous, and so on.

    This means we get into something we call (very expressively)Â `Callback Hell`. And who wants that? It also forms a code pyramid which is not only confusing but also error-prone.

    There are a few ways to come out of (or not get into)Â `callback hell`. The most common one is by using aÂ `Promise`Â orÂ `async`Â function. However, to understandÂ `async`Â functions well, you need to have a fair understanding ofÂ `Promise`s first.

    # **Understanding Promise States**

    Just to review, a promise can be created with the constructor syntax, like this:

    ```jsx
    let promise = new Promise(function(resolve, reject) {
      // Code to execute
    });
    ```

    The constructor function takes a function as an argument. This function is called theÂ `executor function`.

    ```jsx
    // Executor function passed to the
    // Promise constructor as an argument
    function(resolve, reject) {
        // Your logic goes here...
    }
    ```

    The executor function takes two arguments,Â `resolve`Â andÂ `reject`. These are the callbacks provided by the JavaScript language. Your logic goes inside the executor function that runs automatically when aÂ `new Promise`Â is created.

    For the promise to be effective, the executor function should call either of the callback functions,Â `resolve`Â orÂ `reject`. We will learn more about this in detail in a while.

    TheÂ `new Promise()`Â constructor returns aÂ `promise`Â object. As the executor function needs to handle async operations, the returned promise object should be capable of informing when the execution has been started, completed (resolved) or retuned with error (rejected).

    AÂ `promise`Â object has the following internal properties:

    1. `state`Â â€“ This property can have the following values:
    - `pending`: Initially when the executor function starts the execution.
    - `fulfilled`: When the promise is resolved.
    - `rejected`: When the promise is rejected.

    ![https://www.freecodecamp.org/news/content/images/2020/11/states_1.png](https://www.freecodecamp.org/news/content/images/2020/11/states_1.png)

    Promise states

    2. Â `result`Â â€“ This property can have the following values:

    - `undefined`: Initially when theÂ `state`Â value isÂ `pending`.
    - `value`: WhenÂ `resolve(value)`Â is called.
    - `error`: WhenÂ `reject(error)`Â is called.

    These internal properties are code-inaccessible but they are inspectable. This means that we will be able to inspect theÂ `state`Â andÂ `result`Â property values using the debugger tool, but we will not be able to access them directly using the program.

    ![https://www.freecodecamp.org/news/content/images/2020/11/promise_state_inspect.png](https://www.freecodecamp.org/news/content/images/2020/11/promise_state_inspect.png)

    Able to inspect the internal properties of a promise

    A promise's state can beÂ `pending`,Â `fulfilled`Â orÂ `rejected`. A promise that is either resolved or rejected is calledÂ `settled`.

    ![https://www.freecodecamp.org/news/content/images/2020/11/states_2.png](https://www.freecodecamp.org/news/content/images/2020/11/states_2.png)

    A settled promise is either fulfilled or rejected

    ### **How promises are resolved and rejected**

    Here is an example of a promise that will be resolved (`fulfilled`Â state) with the valueÂ `I am done`Â immediately.

    ```jsx
    let promise = new Promise(function(resolve, reject) {
        resolve("I am done");
    });
    ```

    The promise below will be rejected (`rejected`Â state) with the error messageÂ `Something is not right!`.

    ```jsx
    let promise = new Promise(function(resolve, reject) {
        reject(new Error('Something is not right!'));
    });
    ```

    An important point to note:

    > A Promise executor should call only one
    >
    >
    > ```
    > resolve
    > ```
    >
    > ```
    > reject
    > ```
    >
    > ```
    > resolve
    > ```
    >
    > ```
    > reject
    > ```
    >

    ```jsx
    let promise = new Promise(function(resolve, reject) {
      resolve("I am surely going to get resolved!");

      reject(new Error('Will this be ignored?')); // ignored
      resolve("Ignored?"); // ignored
    });
    ```

    In the example above, only the first one to resolve will be called and the rest will be ignored.

    # **How to handle a Promise once you've created it**

    AÂ `Promise`Â uses an executor function to complete a task (mostly asynchronously). A consumer function (that uses an outcome of the promise) should get notified when the executor function is done with either resolving (success) or rejecting (error).

    The handler methods,Â `.then()`,Â `.catch()`Â andÂ `.finally()`, help to create the link between the executor and the consumer functions so that they can be in sync when a promiseÂ `resolve`s orÂ `reject`s.

    ![https://www.freecodecamp.org/news/content/images/2020/11/consumer_executor.png](https://www.freecodecamp.org/news/content/images/2020/11/consumer_executor.png)

    # **How to Use theÂ `.then()`Â Promise Handler**

    TheÂ `.then()`Â method should be called on the promise object to handle a result (resolve) or an error (reject).

    It accepts two functions as parameters. Usually, theÂ `.then()`Â method should be called from the consumer function where you would like to know the outcome of a promise's execution.

    ```jsx
    promise.then(
      (result) => {
         console.log(result);
      },
      (error) => {
         console.log(error);
      }
    );
    ```

    If you are interested only in successful outcomes, you can just pass one argument to it, like this:

    ```jsx
    promise.then(
      (result) => {
          console.log(result);
      }
    );
    ```

    If you are interested only in the error outcome, you can passÂ `null`Â for the first argument, like this:

    ```jsx
    promise.then(
      null,
      (error) => {
          console.log(error)
      }
    );
    ```

    However, you can handle errors in a better way using theÂ `.catch()`Â method that we will see in a minute.

    Let's look at a couple of examples of handling results and errors using theÂ `.then`Â andÂ `.catch`Â handlers. We will make this learning a bit more fun with a few real asynchronous requests. We will use theÂ [PokeAPI](https://pokeapi.co/)Â to get information about PokÃ©mon and resolve/reject them using Promises.

    First, let us create a generic function that accepts a PokeAPI URL as argument and returns a Promise. If the API call is successful, a resolved promise is returned. A rejected promise is returned for any kind of errors.

    We will be using this function in several examples from now on to get a promise and work on it.

    ```jsx
    function getPromise(URL) {
      let promise = new Promise(function (resolve, reject) {
        let req = new XMLHttpRequest();
        req.open("GET", URL);
        req.onload = function () {
          if (req.status == 200) {
            resolve(req.response);
          } else {
            reject("There is an Error!");
          }
        };
        req.send();
      });
      return promise;
    }
    ```

    Example 1: Get 50 PokÃ©mon's information:

    ```jsx
    const ALL_POKEMONS_URL = 'https://pokeapi.co/api/v2/pokemon?limit=50';

    // We have discussed this function already!
    let promise = getPromise(ALL_POKEMONS_URL);

    const consumer = () => {
        promise.then(
            (result) => {
                console.log({result}); // Log the result of 50 Pokemons
            },
            (error) => {
                // As the URL is a valid one, this will not be called.
                console.log('We have encountered an Error!'); // Log an error
        });
    }

    consumer();
    ```

    Example 2: Let's try an invalid URL

    ```jsx
    const POKEMONS_BAD_URL = 'https://pokeapi.co/api/v2/pokemon-bad/';

    // This will reject as the URL is 404
    let promise = getPromise(POKEMONS_BAD_URL);

    const consumer = () => {
        promise.then(
            (result) => {
                // The promise didn't resolve. Hence, it will
                // not be executed.
                console.log({result});
            },
            (error) => {
                // A rejected prmise will execute this
                console.log('We have encountered an Error!'); // Log an error
            }
        );
    }

    consumer();
    ```

    # **How to Use theÂ `.catch()`Â Promise Handler**

    You can use this handler method to handle errors (rejections) from promises. The syntax of passingÂ `null`Â as the first argument to theÂ `.then()`Â is not a great way to handle errors. So we haveÂ `.catch()`Â to do the same job with some neat syntax:

    ```jsx
    // This will reject as the URL is 404
    let promise = getPromise(POKEMONS_BAD_URL);

    const consumer = () => {
        promise.catch(error => console.log(error));
    }

    consumer();
    ```

    If we throw an Error likeÂ `new Error("Something wrong!")`Â Â instead of calling theÂ `reject`Â from the promise executor and handlers, it will still be treated as a rejection. It means that this will be caught by theÂ `.catch`Â handler method.

    This is the same for anyÂ *synchronous*Â exceptions that happen in the promise executor and handler functions.

    Here is an example where it will be treated like a reject and theÂ `.catch`Â handler method will be called:

    ```jsx
    new Promise((resolve, reject) => {
      throw new Error("Something is wrong!");// No reject call
    }).catch((error) => console.log(error));
    ```

    # **How to Use theÂ `.finally()`Â Promise Handler**

    TheÂ `.finally()`Â handler performs cleanups like stopping a loader, closing a live connection, and so on. TheÂ `finally()`Â method will be called irrespective of whether a promiseÂ `resolve`s orÂ `reject`s. It passes through the result or error to the next handler which can call a .then() or .catch() again.

    Here is an example that'll help you understand all three methods together:

    ```jsx
    let loading = true;
    loading && console.log('Loading...');

    // Gatting Promise
    promise = getPromise(ALL_POKEMONS_URL);

    promise.finally(() => {
        loading = false;
        console.log(`Promise Settled and loading is ${loading}`);
    }).then((result) => {
        console.log({result});
    }).catch((error) => {
        console.log(error)
    });
    ```

    To explain a bit further:

    - TheÂ `.finally()`Â method makes loadingÂ `false`.
    - If the promise resolves, theÂ `.then()`Â method will be called. If the promise rejects with an error, theÂ `.catch()`Â method will be called. TheÂ `.finally()`Â will be called irrespective of the resolve or reject.

    # **What is the Promise Chain?**

    The Â `promise.then()`Â call always returns a promise. This promise will have theÂ `state`Â asÂ `pending`Â andÂ `result`Â asÂ `undefined`. It allows us to call the nextÂ `.then`Â method on the new promise.

    When the firstÂ `.then`Â method returns a value, the nextÂ `.then`Â method can receive that. The second one can now pass to the thirdÂ `.then()`Â and so on. This forms a chain ofÂ `.then`Â methods to pass the promises down. This phenomenon is called theÂ `Promise Chain`.

    ![https://www.freecodecamp.org/news/content/images/2020/12/image-105.png](https://www.freecodecamp.org/news/content/images/2020/12/image-105.png)

    Promise Chain

    Here is an example:

    ```jsx
    let promise = getPromise(ALL_POKEMONS_URL);

    promise.then(result => {
        let onePokemon = JSON.parse(result).results[0].url;
        return onePokemon;
    }).then(onePokemonURL => {
        console.log(onePokemonURL);
    }).catch(error => {
        console.log('In the catch', error);
    });
    ```

    Here we first get a promise resolved and then extract the URL to reach the first PokÃ©mon. We then return that value and it will be passed as a promise to the next .then() handler function. Hence the output,

    ```jsx
    https://pokeapi.co/api/v2/pokemon/1/
    ```

    TheÂ `.then`Â method can return either:

    - A value (we have seen this already)
    - A brand new promise.

    It can also throw an error.

    Here is an example where we have created a promise chain with theÂ `.then`Â methods which returns results and a new promise:

    ```jsx
    // Promise Chain with multiple then and catch
    let promise = getPromise(ALL_POKEMONS_URL);

    promise.then(result => {
        let onePokemon = JSON.parse(result).results[0].url;
        return onePokemon;
    }).then(onePokemonURL => {
        console.log(onePokemonURL);
        return getPromise(onePokemonURL);
    }).then(pokemon => {
        console.log(JSON.parse(pokemon));
    }).catch(error => {
        console.log('In the catch', error);
    });
    ```

    In the firstÂ `.then`Â call we extract the URL and return it as a value. This URL will be passed to the secondÂ `.then`Â call where we are returning a new promise taking that URL as an argument.

    This promise will be resolved and passed down to the chain where we get the information about the PokÃ©mon. Here is the output:

    ![https://www.freecodecamp.org/news/content/images/2020/11/image-159.png](https://www.freecodecamp.org/news/content/images/2020/11/image-159.png)

    Output of the promise chain call

    In case there is an error or a promise rejection, the .catch method in the chain will be called.

    A point to note: CallingÂ `.then`Â multiple times doesn't form a Promise chain. You may end up doing something like this only to introduce a bug in the code:

    ```jsx
    let promise = getPromise(ALL_POKEMONS_URL);

    promise.then(result => {
        let onePokemon = JSON.parse(result).results[0].url;
        return onePokemon;
    });
    promise.then(onePokemonURL => {
        console.log(onePokemonURL);
        return getPromise(onePokemonURL);
    });
    promise.then(pokemon => {
        console.log(JSON.parse(pokemon));
    });

    ```

    We call theÂ `.then`Â method three times on the same promise, but we don't pass the promise down. This is different than the promise chain. In the above example, the output will be an error.

    # **How to Handle Multiple Promises**

    Apart from the handler methods (.then, .catch, and .finally), there are six static methods available in the Promise API. The first four methods accept an array of promises and run them in parallel.

    1. Promise.all
    2. Promise.any
    3. Promise.allSettled
    4. Promise.race
    5. Promise.resolve
    6. Promise.reject

    Let's go through each one.

    # **The Promise.all() method**

    `Promise.all([promises])`Â accepts a collection (for example, an array) of promises as an argument and executes them in parallel.

    This method waits for all the promises to resolve and returns the array of promise results. If any of the promises reject or execute to fail due to an error, all other promise results will be ignored.

    Let's create three promises to get information about three PokÃ©mons.

    ```jsx
    const KAKUNA_POKEMONS_URL = 'https://pokeapi.co/api/v2/pokemon/kakuna';

    let promise_1 = getPromise(BULBASAUR_POKEMONS_URL);
    let promise_2 = getPromise(RATICATE_POKEMONS_URL);
    let promise_3 = getPromise(KAKUNA_POKEMONS_URL);
    ```

    Use the Promise.all() method by passing an array of promises.

    ```jsx
    Promise.all([promise_1, promise_2, promise_3]).then(result => {
        console.log({result});
    ```

    # **Can we rewrite the PizzaHub example with Promises?**

    Sure, let's do it. Let us assume that theÂ `query`Â method will return a promise. Here is an example query() method. In real life, this method may talk to a database and return results. In this case, it is very much hard-coded but serves the same purpose.

    ```jsx
    function query(endpoint) {
      if (endpoint === `/api/pizzahub/`) {
        return new Promise((resolve, reject) => {
          resolve({'shopId': '123'});
        })
      } else if (endpoint.indexOf('/api/pizzahub/pizza/') >=0) {
        return new Promise((resolve, reject) => {
          resolve({pizzas: [{'type': 'veg', 'name': 'margherita', 'id': '123'}]});
        })
      } else if (endpoint.indexOf('/api/pizzahub/beverages') >=0) {
        return new Promise((resolve, reject) => {
          resolve({id: '10', 'type': 'veg', 'name': 'margherita', 'beverage': 'coke'});
        })
      } else if (endpoint === `/api/order`) {
        return new Promise((resolve, reject) => {
          resolve({'type': 'veg', 'name': 'margherita', 'beverage': 'coke'});
        })
      }
    }
    ```

    Next is the refactoring of ourÂ `callback hell`. To do that, first, we will create a few logical functions:

    ```jsx
    // Returns a shop id
    let getShopId = result => result.shopId;

    // Returns a promise with pizza list for a shop
    let getPizzaList = shopId => {
      const url = `/api/pizzahub/pizza/${shopId}`;
      return query(url);
    }

    // Returns a promise with pizza that matches the customer request
    let getMyPizza = (result, type, name) => {
      let pizzas = result.pizzas;
      let myPizza = pizzas.find((pizza) => {
        return (pizza.type===type && pizza.name===name);
      });
      const url = `/api/pizzahub/beverages/${myPizza.id}`;
      return query(url);
    }

    // Returns a promise after Placing the order
    let performOrder = result => {
      let beverage = result.id;
       return query(`/api/order`, {'type': result.type, 'name': result.name, 'beverage': result.beverage});
    }

    // Confirm the order
    let confirmOrder = result => {
        console.log(`Your order of ${result.type} ${result.name} with ${result.beverage} has been placed!`);
    }
    ```

    Use these functions to create the required promises. This is where you should compare with theÂ `callback hell`Â example. This is so nice and elegant.

    ```jsx
    function orderPizza(type, name) {
      query(`/api/pizzahub/`)
      .then(result => getShopId(result))
      .then(shopId => getPizzaList(shopId))
      .then(result => getMyPizza(result, type, name))
      .then(result => performOrder(result))
      .then(result => confirmOrder(result))
      .catch(function(error){
        console.log(`Bad luck, No Pizza for you today!`);
      })
    }
    ```
